# Стандарт классификации промптов

Версия: 0.1 | Дата: 2026-05-28

---

## 1. Цель документа

Определить **стандартизированную классификацию промптов**, применимую ко всему
репозиторию `hybrid-Intelligence-lab`, которая позволяет:

1. Осознанно выбирать тип промпта под уровень зрелости проекта и задачу.
2. Тестировать практику «отладки формата промптов» на единых шаблонах.
3. Масштабировать успешные паттерны между проектами.
4. Интегрировать международный опыт prompt engineering и agent design patterns.

**Гипотеза**: матрица «тип промпта × уровень зрелости × сценарий применения»
позволяет БА быстро выбрать адекватный тип промпта без перебора вариантов.

**Ограничения**: классификация основана на промптах, доступных в репозитории
(6 Mango промптов, 3 эксперимента). Прикреплённые документы с 18 промптами
пользователя и примером отладки недоступны для автоматической загрузки; после
их получения матрица должна быть дополнена.

### 2.2 Выбранная основа

**Основной подход**: адаптация OpenAI/Anthropic prompt engineering patterns
с наложением LangChain agent architecture levels и TM Forum AI maturity.

**Обоснование**: OpenAI и Anthropic дают самую практичную и проверенную базу
для классификации промптов по структуре и сложности. LangChain добавляет
agent/multi-agent уровни. TM Forum обеспечивает телеком-контекст. BABOK
связывает промпты с реальными задачами бизнес-аналитика.

## 3. Таксономия промптов (6 осей классификации)

### 3.1 Ось 1: Тип промпта (Prompt Type)

Определяет степень контроля AI над поведением и выводом.

| Тип | ID | Характеристика | Пример |
|-----|----|----------------|--------|
| **Simple** | `PT-1` | Недетерминированный вывод; нет структуры; быстрый результат без гарантий | «Какие фичи есть у облачной АТС?» |
| **Structured** | `PT-2` | Детерминированный вывод; определена роль, алгоритм, правила и формат | Промпт TZ Stats Generator с секциями РОЛЬ/АЛГОРИТМ/ПРАВИЛА |
| **Agent** | `PT-3` | Автоматизация с инструментами; memory, tool use, self-correction | n8n-сценарий: «Прочитай ТЗ → классифицируй → запиши в таблицу» |
| **Multi-Agent** | `PT-4` | Координация нескольких AI-сущностей; orchestration, delegation | Pipeline: «Классификатор → Генератор US → Генератор UC → Reviewer» |

**Источник**: адаптация OpenAI prompt complexity levels + LangChain agent
patterns (chain → agent → multi-agent). Уровни PT-1..PT-4 соответствуют
4 уровням зрелости проекта из issue #79.

### 3.2 Ось 2: Доступ к знаниям (Knowledge Access)

Определяет, откуда промпт получает контекст для генерации.

| Значение | ID | Характеристика | Пример в репо |
|----------|----|----------------|---------------|
| **Standalone** | `KA-S` | Все знания встроены в промпт; не требует внешних источников | `*_simple-2026-05.md` промпты |
| **RAG-enabled** | `KA-R` | Требует подключения внешней базы знаний (`kb/`, `classification.md`) | `*_exp-2026-05.md` промпты |
| **Tool-augmented** | `KA-T` | Использует инструменты для получения данных в реальном времени (API, DB, web) | Будущие agent-промпты с n8n-интеграцией |

**Источник**: Anthropic RAG patterns + LangChain retrieval types.

### 3.3 Ось 3: Режим диалога (Dialog Mode)

Определяет модель взаимодействия между пользователем и AI.

| Значение | ID | Характеристика | Пример в репо |
|----------|----|----------------|---------------|
| **Single-shot** | `DM-1` | Один запрос → один ответ; без уточнений | Simple промпт для быстрой идеи |
| **Multi-turn** | `DM-M` | Диалог с уточнениями, clarification gates, stop-points | User Story Generator с обязательными вопросами |
| **Orchestrated** | `DM-O` | Автоматическая цепочка вызовов; человек подключается только в контрольных точках | Pipeline: classify → generate → validate → approve |

**Источник**: Anthropic conversation patterns + TM Forum intent lifecycle.

### 3.4 Ось 4: Сценарий применения (Consumer)

Определяет, кто является потребителем вывода промпта.

| Значение | ID | Характеристика | Пример |
|----------|----|----------------|--------|
| **Human-in-chat** | `SC-H` | Человек читает и интерпретирует вывод в чате | Все текущие промпты Mango |
| **Agent-consumer** | `SC-A` | Вывод передаётся другому AI-агенту как вход | Use Case Generator получает User Story |
| **API/System** | `SC-S` | Вывод используется системой через API (JSON/YAML schema) | Будущий классификатор с JSON output для n8n |

**Источник**: LangChain composability patterns + TM Forum AI/ML IG1218.

### 3.5 Ось 5: Уровень зрелости проекта (Project Maturity)

Фиксирует уровень инфраструктуры проекта, на котором промпт применяется.

| Уровень | Название | Характеристика | Инфраструктура |
|---------|----------|----------------|---------------|
| **1** | Простой промпт | Быстро, без гарантий | Чат (ChatGPT, Claude) |
| **2** | Системный промпт + артефакты | Предсказуемость, ссылки на `kb/` | Чат + KB + copy-paste промпты |
| **3** | ИИ-агент | Автоматизация, инструменты + память | n8n / LangChain / Claude Agent SDK |
| **4** | Мульти-агентная система | Координация агентов | Orchestration platform + monitoring |

**Источник**: issue #79 (зафиксировано ранее) + TM Forum AI maturity model.

### 3.6 Ось 6: Домен применимости (Domain Scope)

Определяет, специфичен ли промпт для конкретного проекта или универсален.

| Значение | ID | Характеристика | Пример |
|----------|----|----------------|--------|
| **Domain-specific** | `DS-D` | Привязан к конкретному проекту/продукту (термины, KB, taxonomy) | Все Mango промпты с `scope: mango-only` |
| **Universal** | `DS-U` | Применим к любому проекту/домену без адаптации | Шаблон brainstorming, SWOT-analysis |

**Источник**: предложено на основе аудита (все промпты в репо — domain-specific).

## 6. Шаблоны промптов для тестирования «отладки формата»

### 6.1 Шаблон A: Simple Prompt (standalone, single-shot)

**Назначение**: быстрый черновик, brainstorming, explorative Q&A.
**Структура**: `Роль → Задача → Формат вывода → Призыв к действию`.

#### Пример «до отладки» (v0)

```
Придумай сценарии использования облачной АТС для малого бизнеса.
```

**Проблемы v0**: нет роли, нет формата, нет ограничений → вывод
непредсказуемый, может быть 3 пункта или 30, с разным уровнем детализации.

#### Пример «после отладки» (v1)

```
Ты — бизнес-аналитик IT-компании. Предложи 5 сценариев использования
облачной АТС для малого бизнеса (до 50 сотрудников).

Формат: пронумерованный список. Каждый пункт: название (до 10 слов) →
краткое описание (1-2 предложения) → ключевая выгода для бизнеса.
```

#### Чек-лист отладки Simple Prompt

- [ ] Указана роль AI?
- [ ] Задача сформулирована однозначно?
- [ ] Определён формат вывода (список, таблица, текст)?
- [ ] Есть ограничения (количество, длина, аудитория)?
- [ ] Промпт можно скопировать и вставить в любой чат без контекста?

### 6.2 Шаблон B: System Prompt (RAG-enabled, multi-turn)

**Назначение**: нормализация требований, генерация User Story, классификация.
**Структура**: `Роль → Контекст (ссылки на kb/) → Алгоритм → Чек-лист →
Формат + логи`.

#### Пример «до отладки» (v0)

```
Ты помогаешь бизнес-аналитику. Пользователь даст тебе требование из ТЗ,
а ты должен его классифицировать по нашей системе и создать User Story.
Используй taxonomy из classification.md.
```

**Проблемы v0**: нет алгоритма (что делать первым?), нет правил (что нельзя?),
нет формата вывода, нет CTA (что ожидать от пользователя?), нет stop-gates
(когда спрашивать?).

#### Пример «после отладки» (v1)

```
# РОЛЬ
Ты — BA-ассистент. Помогаешь нормализовать сырое требование из ТЗ в
структурированную User Story с привязкой к taxonomy проекта.

# КОНТЕКСТ
Пользователь предоставит:
1. Сырое требование (текст из ТЗ)
2. Выдержку из `classification.md` (taxonomy проекта)

# КАК РАБОТАЕМ
Шаг 1. Определи полноту входных данных (роль, цель, ценность, объект, источник, границы).
Шаг 2. Если данных < 4 из 6 → задай до 3 уточняющих вопросов. **СТОП: жди ответ.**
Шаг 3. Нормализуй терминологию по taxonomy.
Шаг 4. Сгенерируй User Story + Acceptance Criteria.

# ПРАВИЛА
- Не добавляй SLA, даты и API, если их нет во входных данных.
- Не генерируй финальную US без подтверждения пользователем уточнений.
- Помечай предположения как `⚠️ Assumed`.

# ФОРМАТ ВЫВОДА
**Статус**: ready | draft | needs-clarification
**User Story**: As a [role], I want [goal], so that [value].
**Acceptance Criteria**: numbered list
**Mapping**: capability_id, domain, uncertainty
**Вопросы** (если есть): numbered list

# НАЧНЕМ?
Пришли сырое требование и выдержку из classification.md.
```

#### Чек-лист отладки System Prompt

- [ ] Указана роль AI и задача?
- [ ] Определены входные данные (что ожидать от пользователя)?
- [ ] Есть алгоритм (порядок шагов)?
- [ ] Есть правила и ограничения (что нельзя)?
- [ ] Определён формат вывода (поля, типы)?
- [ ] Есть stop-gates (когда AI ждёт подтверждения)?
- [ ] Есть CTA (призыв к действию)?
- [ ] Размер промпта: `_exp` ≤ 1000 символов, `_simple` ≤ 3000 символов?
- [ ] Промпт самодостаточен (standalone) или явно указывает зависимости (RAG)?

---

### 6.3 Шаблон C: Agent Prompt (orchestrated, with tools)

**Назначение**: автоматизация повторяющихся задач с инструментами.
**Структура**: `Роль → Инструменты → Память → Ограничения → Протокол
взаимодействия`.

#### Пример «до отладки» (v0)

```
Ты AI-агент для обработки ТЗ. Читай файлы, классифицируй требования и
записывай результат. Используй classification.md как справочник.
```

**Проблемы v0**: не определены инструменты (какие файлы? как читать?),
нет протокола ошибок, нет ограничений на автономные действия, нет
формата промежуточных результатов.

#### Пример «после отладки» (v1)

```
# РОЛЬ
Ты — BI-агент для автоматической классификации требований из ТЗ.

# ИНСТРУМЕНТЫ
- `read_file(path)`: прочитать файл из репозитория
- `search_kb(query)`: поиск по kb/product-matrix.md
- `write_result(data)`: записать результат в output/

# ПАМЯТЬ
- Храни список обработанных требований в `state.json`.
- При перезапуске продолжай с последнего необработанного.

# КАК РАБОТАЕМ
1. Прочитай следующее необработанное требование из input/.
2. Нормализуй терминологию: search_kb для маппинга на capability.
3. Если confidence < 0.7 → пометь как `needs-human-review`.
4. Запиши результат в output/ в формате YAML.
5. Обнови state.json. Повтори с шага 1.

# ОГРАНИЧЕНИЯ
- Не удаляй и не перезаписывай файлы в input/.
- Не создавай новые capability_id — только маппинг на существующие.
- При 3 последовательных ошибках → остановись и уведоми оператора.

# ПРОТОКОЛ ВЗАИМОДЕЙСТВИЯ
- Человек запускает агента и проверяет output/ по завершении.
- Агент не отправляет сообщений без запроса.
- Логи: output/agent-log-YYYY-MM-DD.md
```

#### Чек-лист отладки Agent Prompt

- [ ] Указана роль агента?
- [ ] Определены инструменты (название, сигнатура, описание)?
- [ ] Определена модель памяти (state, контекст между запусками)?
- [ ] Есть алгоритм работы (шаги)?
- [ ] Есть ограничения на автономные действия (что нельзя)?
- [ ] Определён протокол ошибок (что делать при сбое)?
- [ ] Определён протокол взаимодействия с человеком (когда уведомлять)?
- [ ] Определён формат промежуточных и финальных результатов?
- [ ] Есть логирование действий?

## 8. ❓ Вопросы для согласования

1. **Достаточно ли 6 осей классификации** или стоит добавить/убрать оси?
   Кандидаты на добавление: «язык промпта» (ru/en), «модель AI» (GPT/Claude/Gemini),
   «критичность вывода» (draft/production).

2. **Какой международный стандарт выбрать как основу?**
   В текущей версии предложена комбинация OpenAI/Anthropic + LangChain + TM Forum.
   Альтернатива: взять за основу только TM Forum для более строгой телеком-специфичности.

3. **Какие шаблоны приоритетны для первого теста отладки?**
   Рекомендация: начать с Шаблона B (System Prompt), т.к. он наиболее
   распространён в репозитории и даёт измеримый результат (до/после сравнение).

4. **Нужно ли добавлять ось «домен: телеком/универсальный»?**
   Текущая ось 6 (Domain Scope) разделяет только `domain-specific` vs `universal`.
   Можно расширить: `telecom` / `BA-generic` / `education` / `governance`.

5. **Как поступить с недоступными .docx файлами?**
   Варианты: (a) пользователь добавляет содержимое как `.md` в `research/`;
   (b) пользователь копирует текст в комментарий к issue #79;
   (c) аудит завершается без этих данных (текущий вариант).

6. **Должна ли классификация быть стандартом (`standards/`) или остаться
   исследованием (`research/`)?**
   Текущий статус: `draft` в `research/`. После review и согласования может
   быть повышена до `standards/PROMPT_CLASSIFICATION.md`.
